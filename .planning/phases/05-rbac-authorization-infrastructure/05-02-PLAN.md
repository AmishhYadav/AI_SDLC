## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| principal email → permission set | Resolution decides capability; a wrong/over-broad result is privilege escalation |
| request → database | The resolver reads identity data; it must never write during authorization |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03 | Elevation of Privilege | permission-resolution query | mitigate | Exact-code, data-driven resolution only (D-03); union limited to a non-deleted User (`User.deletedAt: null`) plus non-deleted, non-expired UserRole/Role/RolePermission/Permission rows (D-07); a soft-deleted (deactivated) principal resolves to an empty Set (least-privilege, D-04); no role-name bypass, no hierarchy |
| T-05-04 | Elevation of Privilege / fail-open | unknown or soft-deleted user or empty roles | mitigate | Fail-closed: null user (unknown or soft-deleted) or no active roles ⇒ empty Set (D-04); resolver never throws on absence and never treats an error as allow |
| T-05-05 | Tampering | authorization during request | mitigate | Resolver performs a read-only `findFirst`; no create/update/upsert — no provisioning or write path exists (D-04) |
| T-05-06 | Denial of Service | per-request query cost | mitigate | Single indexed nested query (User.email unique; UserRole/RolePermission indexes) + per-request CLS memoization (D-08); no N+1 |

No package installs occur in this plan — no supply-chain checkpoint required.
