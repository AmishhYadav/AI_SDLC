<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client request to gated handler | Untrusted principal crosses into an authorized operation; the guard is the enforcement point |
| authN result to authZ decision | authorization must derive only from resolved permissions, never from the fact that authentication succeeded |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Elevation of Privilege | authN/authZ conflation | mitigate | Guard decides solely from the resolved Set (RBAC-04, D-06); a valid token with insufficient permissions throws 403, never passes on auth state alone |
| T-05-08 | Elevation of Privilege | guard registration order | mitigate | PermissionsGuard registered strictly after JwtAuthGuard (D-05); order asserted by an automated check; if `request.user` is undefined the guard fails closed to 403 |
| T-05-09 | Elevation of Privilege | AND-match bypass | mitigate | `every()` over required codes (D-02); exact-code membership only, no hierarchy or role-name bypass (D-03) |
| T-05-10 | Information Disclosure | 403 message | mitigate | Fixed generic message constant; errorCode `AUTHZ.PERMISSION_DENIED`; never names the missing permission/role/ID (D-54) |
| T-05-11 | Elevation of Privilege | stub backdoor | mitigate | No `AUTH_MODE`/stub branch in the guard; stub identities resolve permissions from the DB like any principal and get 403 without seeded roles (D-09) |

No package installs occur in this plan — no supply-chain checkpoint required.
