<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS environment → NestJS ConfigModule | Untrusted process env vars cross into validated config at app startup |
| Config module escape hatch → rest of codebase | The no-restricted-properties rule allows process.env in config/ only; all other code must go through AppConfigService |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02 | Tampering | ConfigModule.forRoot validate() callback | mitigate | Use envSchema.parse(env) — throws ZodError on missing/invalid env vars; NestJS catches ZodError at bootstrap and exits non-zero before serving any requests (fail-fast per INFRA-02) |
| T-02-03 | Tampering | process.env direct access outside config/ | mitigate | no-restricted-properties ESLint rule bans process.env in all *.ts files; config/ escape hatch is the only exception; CI gate from Phase 1 enforces this on every push |
| T-02-04 | Denial of Service | Missing DATABASE_URL env var | mitigate | DATABASE_URL is in the required env set (D-07); z.string().min(1) rejects empty string; app cannot start without a valid value — prevents silent runtime Prisma connection failure |
</threat_model>
