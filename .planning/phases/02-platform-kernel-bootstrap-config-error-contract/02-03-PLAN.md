<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Exception handler → HTTP response body | The filter controls what details escape to the client |
| @repo/database → exception filter | Prisma error objects (with meta containing schema details) cross this boundary; the filter must sanitize |
| Inbound x-request-id / traceparent header → req.traceId | Untrusted header value accepted as correlation ID |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Information Disclosure | GlobalExceptionFilter — stack traces in production | mitigate | body['stack'] is only added when !config.isProduction; isProduction reads from Zod-validated NODE_ENV via AppConfigService (never process.env directly); acceptance criterion asserts stack absent when isProduction=true |
| T-02-06 | Information Disclosure | PrismaExceptionFilter — exception.meta leakage | mitigate | PrismaExceptionFilter uses a static PRISMA_HTTP_MAP with fixed message strings; exception.meta (which contains table/field/constraint names) is never referenced in the response body; acceptance criterion greps for exception.meta in filter source |
| T-02-07 | Information Disclosure | PrismaExceptionFilter — unknown Prisma error code | mitigate | Unmapped codes fall through to status=500, errorCode=PLATFORM.INTERNAL_ERROR, message='Database operation failed' — no code or meta forwarded |
| T-02-08 | Spoofing | CorrelationIdMiddleware — inbound x-request-id accepted | accept | Per D-01: traceId is correlation-only, not used for authentication or access control. No security decision is based on traceId value. Accepting inbound correlation ID is the designed behavior for distributed tracing. |
</threat_model>
