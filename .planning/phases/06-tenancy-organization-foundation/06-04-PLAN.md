      let prisma: PrismaService;
      let orgAId: string;
      let orgBId: string;
      let userAEmail: string;
      let userBEmail: string;
      // Also create a user with INVITED (non-ACTIVE) membership for test (e)

        userAEmail = 'tenant-a@isolation.test';
        userBEmail = 'tenant-b@isolation.test';
        invitedUserEmail = 'tenant-invited@isolation.test';
        creatorEmail = 'tenant-creator@isolation.test';

        // Pre-cleanup: idempotent cleanup of any leftover fixtures
        for (const email of [userAEmail, userBEmail, invitedUserEmail, creatorEmail]) {

            await prisma.organizationMember.deleteMany({ where: { userId: u.id } });
            await prisma.user.delete({ where: { id: u.id } });
          }
        }

        await prisma.organization.deleteMany({ where: { name: { in: ['Isolation Org A', 'Isolation Org B', 'Created Via Api'] } } });

    Title: ADR-001 — Tenant Enforcement Mechanism: Prisma Client Extension
    Date: 2026-07-03
    Status: Accepted

    Context section: This platform is a multi-tenant SaaS system with a shared PostgreSQL schema and multiple organizations accessing the same database. The schema is FROZEN for this milestone (no DDL changes permitted). Connection pooling (PgBouncer or equivalent) is a deployment requirement. The tenant context (organizationId) is available per-request via NestJS CLS (AsyncLocalStorage). We need a mechanism to enforce organization-level data isolation at the database access layer so that no application code path can accidentally query another tenant's data. The selected mechanism must satisfy: (1) no DDL required, (2) connection-pooling-safe, (3) testable with the application's existing test harness.

    Decision section: The tenant enforcement mechanism is a Prisma client extension ($extends) that wraps the existing singleton PrismaService. The extension registers an $allModels.$allOperations query hook via a closure over ClsService. At query execution time the hook reads organizationId from the request's AsyncLocalStorage store and auto-injects WHERE organizationId = {orgId} AND deletedAt IS NULL into eligible operations on the ORG_SCOPED_MODELS whitelist. The extension is fail-closed (D-08): if organizationId is undefined in CLS when a scoped-model query runs, the hook throws ForbiddenException(TENANT.NO_ORG_CONTEXT) rather than proceeding without the scope. Implementation: packages/backend/src/tenancy/tenanted-prisma.service.ts. The two-organization integration test (app.integration.spec.ts describe 'Tenant Isolation (real DB)') is the primary acceptance gate for this decision.

    Alternatives Considered section:
    1. PostgreSQL Row-Level Security (RLS) — REJECTED this phase: requires DDL (CREATE POLICY statements) which conflicts with the frozen schema constraint. Also requires SET LOCAL app.tenant_id = '{id}' per transaction, which pins connections in a session-variable-aware pooler (PgBouncer in transaction mode cannot reset SET LOCAL). Recorded as a possible future defense-in-depth layer on top of the Prisma extension if additional security hardening is needed.
    2. Prisma $use middleware — REJECTED: deprecated in Prisma 5, removed in Prisma 6. The $extends API is the only supported extension mechanism in Prisma 6.19.3.
    3. Request-scoped PrismaClient instance per tenant — REJECTED: creates a new connection per request, exhausting the connection pool under load and eliminating connection reuse benefits.

    Consequences section: (Positive) No DDL required; schema remains unchanged. Connection pool efficiency preserved. Enforcement is testable with the existing Vitest + real-DB integration test harness. The extension creates a lightweight wrapper around the singleton client with no per-request overhead for connection management. (Negative/Mitigated) Enforcement lives in application code — the isolation test (TENANT-06) is the primary safety gate rather than a DB-enforced row security policy. PostgreSQL RLS remains as a future defense-in-depth option to add DDL-based enforcement behind the application layer. Operations that bypass TenantedPrismaService (raw PrismaService injection) require explicit documentation and are auditable via codebase search. The findUnique/findUniqueOrThrow operations are excluded from auto-injection due to Prisma's type-level unique-key constraint enforcement; repositories must use findFirst for org-scoped lookups.
